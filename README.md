## 트랜잭션 검증을 위한 동시 요청 클라이언트 구현 및 동시성 문제 제어 프로젝트

### 문제 정의

- **테스트 환경 부재 문제:** 주문이 많아질 경우 “따닥 이슈”나 트랜잭션 경합이 발생할 수 있으나, 이를 **지속적·반복적으로 검증할 수 있는 테스트 환경**이 존재하지 않는 문제
- **실제 동시성 문제:** 테스트 환경을 구축한 뒤, 다수의 동시 요청에서 **재고 차감 누락(race condition)** 현상이 발생하는 문제.

### 시도 및 해결 과정

1. **테스트 환경 구축**
    - JUnit 내에서 동시 요청을 시뮬레이션할 수 있도록 멀티스레드 기반의 동시성 클라이언트를 직접 구현 (`ExecutorService` + `CountDownLatch`).
    - 이를 통해 **프로젝트 내부에서 반복 가능하게 동시성 이슈를 재현·검증**할 수 있는 기반을 마련했습니다.
2. **문제 확인**
    - 100건 주문 요청 시 평균 27건만 반영되고 73건이 누락되는 race condition 발생.
3. **문제 해결**
    - 주문 시 재고 조회 구간에 **DB Lock**을 적용하여 동일 데이터 동시 접근을 차단.
    - 결과적으로 요청 건수와 실제 재고 차감 수가 일치하도록 개선.

### 결과 및 성과

- **테스트 기반 확보:** 동시성 이슈를 지속적·반복적으로 검증할 수 있는 환경을 구축.
- **정합성 보장:** race condition 문제를 DB Lock을 통해 해결, 트랜잭션의 안정성 확보.
- **역량 강화:** 단순 기능 구현을 넘어, 서비스 운영 단계에서 발생할 수 있는 동시성 문제를 사전에 발견·검증·해결할 수 있는 능력 배양.
